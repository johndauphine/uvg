# Release Process

UVg releases are cut from `master` by a maintainer. `Cargo.toml`, the
top-level package entry in `Cargo.lock`, and crates.io use the bare `X.Y.Z`
version. Git tags and GitHub Releases use `vX.Y.Z`, and workflow-generated
archive names include that tag name, such as `uvg-vX.Y.Z-<target>.tar.gz`.
All of these identifiers must point to the same release version.

The current release automation is `.github/workflows/release.yml`. Pushing a
`v*` tag builds the release binary for Linux x86_64, Linux aarch64, macOS
aarch64, and Windows x86_64, packages each archive with `README.md`, both
licenses, and `NOTICE`, then uploads the archives and SHA-256 checksum files to
GitHub Releases.

## Versioning Policy

UVg follows Semantic Versioning: `MAJOR.MINOR.PATCH`.

The public compatibility surface includes:

- CLI commands, flags, environment-variable behavior, exit status conventions,
  and accepted database URL forms.
- SQLAlchemy model output for supported schema inputs.
- DDL output, migration revision file format, `uvg_version` semantics,
  `--out-dir` layout, and migration apply/downgrade behavior.
- Published binary archive names and the crates.io package.

Use a major release for incompatible changes, including removed or renamed CLI
flags, changed defaults that alter existing workflows, incompatible migration
file format or `uvg_version` behavior, or intentional generated-output changes
that are not fidelity or correctness fixes.

Use a minor release for backward-compatible functionality, including new CLI
flags, new dialect coverage, new metadata support, additional migration
features behind compatible behavior, or generated-output improvements that only
affect newly supported schema features.

Use a patch release for bug fixes, documentation fixes, dependency updates, and
generated-output or migration-output corrections where the previous output was
incorrect, invalid, unsafe, or inconsistent with documented behavior. If a patch
release changes generated bytes or migration SQL for existing inputs, call that
out explicitly in `CHANGELOG.md`.

## Changelog

`CHANGELOG.md` is the source for release notes before a tag is cut. GitHub may
generate additional notes from merged pull requests, but the changelog should
carry the maintainer-written compatibility notes.

For each release:

- Move `Unreleased` entries into a new `vX.Y.Z - YYYY-MM-DD` section.
- Include any breaking changes, generated-output changes, and migration-output
  changes even when they are bug fixes.
- Link notable docs, test fixture, or release-process changes under `Release`.
- Keep `Unreleased` present for the next cycle.

## Release Checklist

Prepare the release in a normal pull request:

- Confirm `master` is green for `Test`, `cargo deny`, and `CRM 9-pair matrix`.
- Pick the next version from the versioning policy.
- Update `Cargo.toml`.
- Run `cargo check` once if needed to refresh the root package version in
  `Cargo.lock`, then include the lockfile change.
- Update `CHANGELOG.md`.
- Review user-facing docs for version-specific examples.
- Run the validation commands below.
- Merge the release-prep PR into `master`.

Validation commands:

```bash
cargo fmt --all -- --check
git diff --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
cargo deny --all-features --locked check advisories licenses bans sources
cargo package --locked
cargo publish --locked --dry-run
```

Run the CRM matrix locally when the release includes dialect, DDL, migration, or
introspection changes and the database containers are available:

```bash
cargo build --release
./testdata/crm/run_matrix.sh --strict
```

## Cutting The Release

After the release-prep PR is merged and `master` is current:

```bash
export UVG_VERSION=X.Y.Z
git fetch origin
git checkout master
git pull --ff-only origin master
git tag -a "v${UVG_VERSION}" -m "uvg ${UVG_VERSION}"
git push origin "v${UVG_VERSION}"
```

Wait for the GitHub Release workflow to finish. Verify the release contains
these files for each target:

- `uvg-vX.Y.Z-x86_64-unknown-linux-gnu.tar.gz`
- `uvg-vX.Y.Z-aarch64-unknown-linux-gnu.tar.gz`
- `uvg-vX.Y.Z-aarch64-apple-darwin.tar.gz`
- `uvg-vX.Y.Z-x86_64-pc-windows-msvc.zip`
- one matching `.sha256` file per archive

Download and verify the checksums before publishing elsewhere:

```bash
gh release download "v${UVG_VERSION}" --pattern "uvg-*"
shasum -a 256 -c *.sha256
```

## Artifact Integrity

Every release archive must have a matching SHA-256 checksum generated by the
release workflow. The checksum files let users detect corrupted or substituted
downloads when they already trust the GitHub Release page. Checksums do not, by
themselves, prove independent signer identity.

Current signing and provenance decision: UVg does not publish detached
signatures, keyless signatures, or SLSA provenance attestations yet. Do not
describe release artifacts as signed or provenance-backed until the release
workflow actually creates and uploads those materials. Revisit this decision
before a `2.0.0` release or before adding distribution channels that require
attestations.

## Crates.io Publishing

Current automation decision: crates.io publishing remains a manual maintainer
step. Do not publish automatically from release tags until the project has a
workflow that can coordinate crates.io publish status with GitHub Release asset
status and avoid partially published releases.

After the GitHub Release workflow succeeds and checksums verify, publish from
the same `master` commit used by the tag:

```bash
git checkout master
git pull --ff-only origin master
git rev-parse HEAD
git rev-list -n 1 "v${UVG_VERSION}"
cargo publish --locked
```

The two git commands must print the same commit. If they do not, stop and
resolve the mismatch before publishing.

## Rollback

Before crates.io publish, a bad tag or failed artifact build can be withdrawn by
deleting the GitHub Release and tag, fixing the release-prep commit, and pushing
a corrected tag. If the release may already have been consumed, prefer a new
patch version instead of reusing the same version.

After crates.io publish, treat the version as immutable. Do not move the tag or
replace archives for the same public version. For a serious release problem:

- Mark the GitHub Release as withdrawn or add a prominent warning to its notes.
- Yank the crates.io version if users should not select it for new installs.
- Fix forward with the next patch version.
- Document the failure and replacement version in `CHANGELOG.md`.

For a checksum mismatch, assume the affected asset is invalid until proven
otherwise. Remove the GitHub Release before crates.io publish, or withdraw the
release and issue a patch release after crates.io publish.
